Siempre que se haga pull hay quer revisar si hay conflictos, de ser así se deben resolver inmediatamente y luego hacer commit seguido de push.

Cuando haya conflictos, no resolverlos usando mine o their, mejor usar la opción de “Edit conflicts“ de Tortoise Git, y determinar como se integran a detalle las modificaciones del código.

En la descripción la primer línea de texto debe tener un resumen consistente en el módulo o identificador del código que se cambió y que fue lo que se cambió. Pueden haber más lineas de texto debajo que pueden incluso copiarse de las tareas de trello. Revisar ortografía y redacción.

GIT Cuando se esta en hotfix o en una rama de master, jalar frecuentemente lo de master (hacer merge de master), en especial antes de hacer push.

GIT Cuando se está en feature o en una rama derivada de develop, jalar frecuentemente lo de develop (hacer merge de develop), en especial antes de hacer push.
